1d:T28c1,<h1 id="">개요</h1><p>현업에서도 가끔 RAG 얘기를 많이 들었다.(백엔드의 영역으로 직접적인 관계는 없었으나..)</p>
<p>대강 어떤 형태인진 이해하고 있었으나 자세히 알아볼 기회가 스스로에게 생겼다.</p>
<p>LLM 을 활용한 개발을 좀 해보다 보니, 개선 적용을 하려니 RAG가 필요해 보였다. 일단 더 알아보고 앞으로 어떤 방향으로 활용해 볼 수 있을지 생각해보기위해, Langchain 공식 문서에서 소개하는 RAG에 대해서 정리해보고 내 맘대로 정리해보자.</p>
<p>일단 RAG는 Retrieval-Augmented Generation (검색 증강 생성) 을 뜻한다.</p>
<h1 id="rag-">RAG의 핵심 장점</h1><p>RAG를 사용하면 얻을 수 있는 장점들이 여럿 있는데, 정리해보면 다음과 같다.</p>
<ol>
<li><strong>Hallucination 감소</strong>: 실제 문서를 기반으로 답변하기 때문에 LLM이 만들어내는 환각(거짓 정보) 현상을 줄일 수 있다.</li>
<li><strong>최신 정보 활용</strong>: LLM의 학습 cutoff 이후의 데이터도 문서로 제공하면 활용할 수 있다.</li>
<li><strong>출처 추적</strong>: 답변의 근거가 되는 문서를 같이 제공할 수 있어서 신뢰성을 높일 수 있다.</li>
<li><strong>도메인 특화</strong>: 기업 내부 문서나 전문 지식을 학습 없이도 활용할 수 있다.</li>
</ol>
<p>결국 LLM의 한계를 외부 지식으로 보완하는게 RAG의 핵심이라고 볼 수 있다.</p>
<h2 id="retrievers">Retrievers</h2><p>구조화 되지 않은 쿼리가 어떤 주어진 문서를 반환하도록 하는 인터페이스 말그대로 검색기이다.</p>
<p><em>벡터 저장소(이후 설명)들을 리트리버로 캐스팅하여 벡터 저장소를 검색할 수 있도록 만들 수 있다.</em></p>
<h1 id="retrival">Retrival</h1><p>LLM은 훈련한 데이터와 가지고있는 유한한 컨텍스트를 기반으로 답변을 만든다.
이것은 답변의 확장이 불가능한 구조를 뜻하기도 한다.</p>
<p>따라서 지식 기반을 만들어주고 문서 또는 구조화된 데이터로서 제공하여 LLM이 참고하여 답변할 수 있도록 개선 할 수 있다.</p>
<p>따라서 이렇게 <strong>검색한 결과</strong>와 <strong>LLM의 생성</strong>이 결합된 RAG를 만들어 더 좋은 검색 결과를 만들어 낼 수 있게 된다.</p>
<h1 id="-2">파이프라인</h1><p>검색의 파이프라인을 간단하게 표현하면 다음과 같다.</p>
<h3 id="-">인덱싱 과정</h3><ol>
<li><strong>문서 로딩</strong>: PDF, 웹페이지, Markdown 등 다양한 형식의 문서를 불러온다.</li>
<li><strong>텍스트 추출</strong>: 문서에서 텍스트를 구조화된 형태로 변환한다.</li>
<li><strong>청킹(Chunking)</strong>: 긴 문서를 적절한 크기로 분할한다. 보통 200~500 토큰 정도로 나눈다. 너무 작으면 문맥이 사라지고, 너무 크면 검색 정확도가 떨어지기 때문이다.</li>
<li><strong>임베딩</strong>: 텍스트를 고차원 벡터(숫자 배열)로 변환한다. 의미가 비슷한 텍스트는 벡터 공간에서 가까이 위치하게 된다.</li>
<li><strong>벡터 저장</strong>: Vector DB(Pinecone, Weaviate, Chroma 등)에 임베딩 벡터를 저장한다.</li>
</ol>
<h2 id="--2">검색 및 생성 과정</h2><ol>
<li><strong>쿼리 임베딩</strong>: 사용자의 질문을 벡터로 변환한다.</li>
<li><strong>유사도 검색</strong>: 코사인 유사도 같은 방법으로 질문과 가장 유사한 상위 K개 문서를 찾는다.</li>
<li><strong>컨텍스트 구성</strong>: 검색된 문서들을 LLM 프롬프트에 추가한다.</li>
<li><strong>답변 생성</strong>: LLM이 컨텍스트를 참고하여 답변을 생성한다.</li>
</ol>
<h1 id="rag--2">RAG 설계 구조</h1><p>아래 형태들은 Langchain 문서에서 발췌해온 대표적인 구조들이다.</p>
<p>실제로는 훨씬 더 많은 패턴들이 연구되고 있고, 정답이 있는게 아니라 내가 만들고자 하는 서비스에 제일 적절한 방식을 선택하고 부족한 영역을 채우는 식으로 설계하는게 중요해보인다.</p>
<p>결국 아래 있는 형태들도 정답도 아니고 대표적인 예시 정도로 봐야하는게 옳아보인다.</p>
<h2 id="naive-rag-basic-rag">Naive RAG (Basic RAG)</h2><p>가장 기본적인 형태로, 생성 전에 만들어놓은 문서만 가지고 검색한다.</p>
<pre><code class="hljs language-bash">사용자 질의 -&gt; 문서 검색(Retrieve Documents) -&gt; LLM에서 답변 생성 -&gt; 답변 제공
</code></pre><p>이미 존재하는 문서를 기반으로 했을 때 이점이 있고 빠르다.</p>
<p><strong>장점</strong>: 구현이 간단하고 빠르다.
<strong>단점</strong>: 검색 품질이 좋지 않으면 답변도 좋지 않다.</p>
<p>ex) FAQ bot, Document bot</p>
<h2 id="advanced-rag">Advanced RAG</h2><p>Naive RAG에 여러 최적화 기법을 추가한 형태다.</p>
<ul>
<li><strong>쿼리 최적화</strong>: 사용자의 질문을 검색에 적합하게 재작성한다.</li>
<li><strong>Re-Ranking</strong>: 검색된 문서들의 순위를 재조정해서 더 관련성 높은 문서를 선택한다.</li>
<li><strong>문서 필터링</strong>: 검색 결과 중에서 품질이 낮은 문서를 걸러낸다.</li>
</ul>
<p>실제 프로덕션에서는 이런 기법들을 조합해서 사용하는게 일반적인 것 같다.</p>
<h2 id="agentic-rag">Agentic RAG</h2><p>Retrieval을 어느 시기에 할지 LLM이 스스로 결정하고 활용한다.</p>
<p>빠르진 않더라도 유연한 답변을 생성하기 좋고 좋은 답변을 얻어낼 수 있도록 할 수 있다.</p>
<p>만약 답변이 적절하지 않다고 판단한다면 좋은 답변을 얻기위해 다시 돌아갈 수도 있다.</p>
<div class="mermaid-placeholder" data-mermaid-id="mermaid-0"></div>

<p><strong>장점</strong>: 복잡한 질문에 대해 더 정확한 답변을 만들어낼 수 있다.
<strong>단점</strong>: 느리고 비용이 많이 든다.</p>
<p>ex) Research Assistants, Agent Tool 활용 서비스</p>
<h2 id="hybrid-search-rag">Hybrid Search RAG</h2><p><strong>Sparse Search(BM25)</strong> 와 <strong>Dense Search(Vector Embeddings)</strong> 를 결합한 방식이다.</p>
<ul>
<li><strong>Sparse Search (BM25)</strong>: 키워드 기반 검색으로, 정확한 용어 매칭에 강하다. 전통적인 검색 방식이라고 생각하면 된다.</li>
<li><strong>Dense Search</strong>: 임베딩 기반 의미 검색으로, 문맥적 유사도를 찾는데 강하다. 같은 의미지만 다른 표현을 쓴 문서도 찾아낸다.</li>
<li>두 방식의 결과를 가중치 기반으로 Re-Ranking하여 최적의 문서를 선택한다.</li>
</ul>
<p>각 검색 방식의 장점을 합쳐서 더 좋은 검색 결과를 만들어 낸 다음 다음 스텝으로 가기 위해 계속해서 Retrieval 하거나 LLM의 결과 생성을 이루도록 한다.</p>
<div class="mermaid-placeholder" data-mermaid-id="mermaid-1"></div>

<p><strong>장점</strong>: 각 검색 방식의 단점을 서로 보완할 수 있다.
<strong>단점</strong>: 두 가지 검색을 다 수행해야 하므로 느리고 복잡하다.</p>
<h2 id="modular-rag">Modular RAG</h2><p>필요에 따라 모듈을 조합하는 방식이다.</p>
<p>위에 설명한 여러 기법들을 레고 블록처럼 조합해서 사용할 수 있다. 실제 서비스에서는 이런 방식으로 구현하는게 유연하고 좋아보인다.</p>
<h1 id="-advanced-rag-">최신 Advanced RAG 기법</h1><p>좀 더 찾아보니 아래 내용들은 조금 더 정리된, RAG로 검색하면 언급되고있는 기법? 구조들인 듯 하다.</p>
<h2 id="self-rag">Self-RAG</h2><p>LLM이 스스로 검색이 필요한지 판단하고, 생성된 답변을 자체 평가한다.</p>
<p>답변의 품질이 낮다고 판단하면 다시 검색하거나 재생성을 시도한다. 사람처럼 자기 답변을 검토하는 느낌이랄까.</p>
<h2 id="corrective-rag-crag">Corrective RAG (CRAG)</h2><p>검색 결과의 품질을 평가하고, 품질이 부족하면 웹 검색 등 추가 소스를 활용한다.</p>
<p>벡터 DB에 관련 문서가 없거나 품질이 낮을 때 외부에서 찾아오는 방식이다.</p>
<h2 id="hyde-hypothetical-document-embeddings">HyDE (Hypothetical Document Embeddings)</h2><p>사용자의 질문을 바로 검색하는게 아니라, 질문에 대한 가상의 이상적인 답변을 먼저 생성한 다음 그걸로 검색한다.</p>
<p>쿼리와 문서 사이의 비대칭 문제를 해결하는 재밌는 접근이다. &quot;이런 답변이 있을 것 같은데&quot; 하고 먼저 만들어보고 그것과 비슷한 문서를 찾는 방식.</p>
<h2 id="graphrag">GraphRAG</h2><p>지식 그래프와 결합하여 구조화된 관계 정보를 활용한다.</p>
<p>단순히 텍스트 유사도만 보는게 아니라, 개체 간의 관계나 구조를 고려해서 검색한다. 복잡한 도메인 지식을 다룰 때 유용해보인다.</p>
<h2 id="long-rag">Long RAG</h2><p>작은 청크 단위가 아니라 긴 문서 단위로 검색하여 문맥을 보존한다.</p>
<p>청킹으로 인해 문맥이 끊기는 문제를 해결하려는 시도다. Long Context를 지원하는 최신 LLM들과 잘 맞는 접근인 것 같다.</p>
<h1 id="-3">마무리</h1><p>RAG는 계속 발전하고 있는 분야인 것 같다. 위에서 설명한 여러 아키텍처들 중 일부는 Langchain 같은 프레임워크에서 제시하는 대표적인 패턴들이고, 실제로는 훨씬 더 많은 변형과 최적화 기법들이 연구되고 있다. (정답이 없는 부분)</p>
<p>요즘은 여러 문서에서 더 발전된 형태들이 많이 발표되고 나오고 있는데, 알아두면 좋을 것 같지만 실제로 내가 만들어보고 싶은게 나온다면 적절하게 찾아보고 제일 알맞은 용도와 부족한 점에 맞게 추가 개선해 나가는것도 재미있는 경험이 될 것 같다.</p>
<p>RAG를 찾다보면 vector store 이야기도 자연스레 접할 수 있다. 심지어 여기도 다양하다. 채용 공고나 현업에서 어떤 친구들로 자주 쓰이는지 보고 한번 나도 따라해보면 될 것 같아 진행해보는 중이다. 현업이 아니다 보니 시간이 부족한게 아쉽다.</p>
