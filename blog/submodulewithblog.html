Git Submodule로 블로그 소스코드는 Private, 배포는 Public으로 관리하기

2025년 10월 5일

소스코드는 숨기고, 결과물만 공개하고 싶다..!

블로그를 만들다보면 GitHub Pages로 배포하는 경우가 많습니다. 그런데 문제가 하나 있습니다.

소스코드가 모두 공개된다는 점

물론 오픈소스로 공개하는 것도 좋지만, 블로그 소스코드에는 개인적인 설정이나 실험 중인 코드, 또는 아직 완성되지 않은 초안 글들이 포함되어있을 수 있습니다.

배포된 결과물만 공개하고, 소스코드는 private으로 관리하고 싶었습니다.

Git Submodule

Git Submodule을 활용하면 이 문제를 해결할 수 있어보였습니다. 예전엔 자주 썻는데, 모노레포로 워크스페이스를 관리하다보니 최근에는 잘 활용한적이 없네요.

아무튼 이런식으로..

  1. 블로그 소스코드를 담은 private 저장소 (예: my-blog-private)
  2. 빌드된 정적 파일들을 담을 public 저장소 (예: username.github.io)
  3. private 저장소 내부에 public 저장소를 submodule로 연결

이렇게 하면 소스코드는 private으로 관리하면서, 빌드 결과물만 public 저장소에 푸시할 수 있습니다.

실제 구조 살펴보기

실제 디렉토리 구조에서 중요한 것만 봅시다..

디렉토리 구조

blog-source/          (private 저장소)
├── ...
├── out/              (public 저장소 - submodule)
│   └── .git/
└── package.json

Submodule 설정

.gitmodules 파일에 submodule을 등록해둡니다

[submodule "out"]
	path = out
	url = git@github.com:내공개레포.git
	branch = deploy

out 디렉토리가 실제 public 저장소를 가리키고 있습니다. 이제 빌드 결과물을 이 디렉토리에 넣고 푸시하면 됩니다

배포 과정

기본적으로 다음과 같은 흐름입니다

  1. 빌드 - Next.js를 정적 사이트로 빌드
  2. 복사 - 빌드 결과물을 out 디렉토리(submodule)로 복사
  3. 푸시 - out 디렉토리에서 public 저장소로 커밋 & 푸시

저는 이 과정도 쉘 스크립트로 자동화를 해두었습니다. 귀찮으면 안되잖아요?

Private 저장소의 변경사항과 Public 저장소(submodule)의 변경사항을 별도로 관리하면 됩니다

마치며

Git Submodule을 사용하면 소스코드는 private으로, 배포는 public으로 관리할 수 있습니다.

물론 GitHub Actions나 Vercel 같은 다른 방법들도 있지만, 저장소를 명확하게 분리하고 싶다면 Submodule도 괜찮은 선택지입니다.
