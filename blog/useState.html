useState를 직접 구현하라는 얘기를 들었어요

2025년 10월 7일

개요

친구랑 대화하다가 useState를 직접 구현해보았냐는 이야기를 들었다

React의 업데이트내용은 프론트엔드 개발자로서 쫓아가야하는 필수 코스기도 하다보니 읽고나서 대강대강 이렇겠거니 생각은 하고있었으나 막상 만들자고 하면 또 분명 머뭇거릴거다. 그래서 한번 복기해보기.(사실 뭔가 끄적이고 싶었어)

물론 제일 좋은것은 직접 구현코드를 쫓아가보는게 좋고 요즘엔 https://deepwiki.com/ 와 같은 서비스도 있어서, 참고하면 더 좋은것같다.

"직접 구현해보았느냐" 라는 맥락에는 왠지 아래와 같은 뜻이 있는것같다.(아닐수도있음.. 그냥 개인적인 의견입니다ㅋ)

  • 내부 구조를 이해하고있는가
  • 상태 변화에 따른 렌더 타이밍을 만들어볼 수 있을까
  • 그 렌더 타이밍을 어떻게 최적화 할 수 있는가

등등이 아닐까 싶었다.

애초에 구현을 모두 하는건 현실적이지않다. 그렇다면 역시 이런 흐름을 만들 수 있는지 이해하면서 생각, 활용할 수 있는가가 중요한것같다.

차근차근

단계별로 구현하면서 이해하는건 항상 이해하는데 도움이 된다고 생각하는 편이라

단순 버전을 먼저 코드로 그려보고 이해해보자. 컴포넌트는 편의상 한개만 존재한다고 생각해두자. 지금은 상태 업데이트를 관리하는 그 흐름을 이해하는 과정에 집중해보자

1단계: 단순버전

격리된 상태에 useState에 기본 인터페이스만 구현되어있는 상태다.

상태가 변경된 후에는 당연히 render 함수가 실행되어야할 것이다. 일단 눈에 잘 보이게 render 를 주입받도록 해두자.

지금은 모든 상태 업데이트마다마다 대상이 렌더가 되는 형태이다.

let state;

function useState(initialValue, render) {
  if (state === undefined) {
    state = initialValue;
  }

  const setState = newValue => {
    state = newValue;
    render();
  };

  return [state, setState];
}

2단계: 상태가 여러개가 될 수 있음을 고려하자

이번엔 states를 배열로 만들어서 컴포넌트에 상태를 여러개 넣을 수 있다는 가정이다.

let states = [];
let currentIndex = 0;

function useState(initialValue, render) {
  const index = currentIndex; // 클로저로 인덱스 캡처

  // 최초 렌더링 시에만 초기화
  if (states[index] === undefined) {
    states[index] = initialValue;
  }

  const setState = newValue => {
    states[index] = newValue;
    currentIndex = 0; // 다음 렌더링을 위해 인덱스 리셋
    render();
  };

  currentIndex++; // 다음 useState를 위해 증가
  return [states[index], setState];
}

3단계: 렌더링이 자주 일어나지 않게 방지하자

단순 값비교로, 렌더가 일어나면 안되는 상황을 만들어주었다.

let states = [];
let currentIndex = 0;

function useState(initialValue, render) {
  const index = currentIndex;

  if (states[index] === undefined) {
    states[index] = initialValue;
  }

  const setState = newValue => {
    // 값이 실제로 변경되었을 때만 렌더링
    if (newValue !== states[index]) {
      states[index] = newValue;
      currentIndex = 0;
      render();
    }
  };

  currentIndex++;
  return [states[index], setState];
}

4단계: 배치 업데이트 형태로 렌더링을 좀더 좁혀주자

이번엔 스케쥴링을 추가했다. 값에 변화가 있다면 스케쥴링에 추가하고, 많은 업데이트가 동시에 발생했을때 그 렌더 횟수를 줄이는 역할을 위함이다.

여기서 마이크로태스크 큐 관련된 내용이 나오는데 잘 정리된 문서가 하나 있으니 참고

// 3단계 코드에서 해당 함수를 추가하고 render 의 실행을 위임한다고 생각하고 간략하게 보자.

let isScheduled = false;
let scheduledRender = null;

function scheduleRender(render) {
  scheduledRender = render;

  if (!isScheduled) {
    isScheduled = true;
    queueMicrotask(() => {
      isScheduled = false;
      currentIndex = 0;
      if (scheduledRender) {
        scheduledRender();
        scheduledRender = null;
      }
    });
  }
}

그외..

여기서 더 나아가면 분명 나올만한 요소는 다음과 같을 것 같다.

  • 상태를 업데이트 큐 공간을 만들어서 업데이트를 한번에 진행하기(마이크로 태스크 큐에 상태에 대한 실질적 업데이트를 몰아서 하기)
  • 우선순위 기반으로 스케쥴링해서 렌더링을 조금 더 타이트하게 관리할 수 있도록 만들기

알아두면 좋은 이유는 뭘까

결국 상태 관리의 기본인 useState의 흐름에 대해 이해하면 얕은 깊든 분명 도움이 될 것이라고 생각한다.

이러한 흐름을 이해하고 구현되어있는 내용에 대해 이해하기 시작하면 어떻게 하면 좀더 좋은 UX를 만들기위해 최적화를 할 수 있을지 아이디어가 떠오를 수 있게 된다.

이뿐만 아니라 React에서 위임되어있는 많은 과정에서 React는 각각에 맞는 최적화 기법을 사용하고있을 것이다.

특히 이런 요소들에 대한 이해는 useTransition이나 useDeferredValue 같은 다양한 훅들을 추후 적재적소에 사용할 수 있는 생각으로 이어질 수 있다고 생각한다.

하지만 아직 안써봣어도 상관없다. 어차피 훅들은 API일 뿐이고 이러한 요소들은 React가 매지컬하게 하고있는 그러한 기법 사이에서 사용자가 느낄 수 있는 애로사항들을 API로 접근할 수 있게 알차게 뚫어둔 요소라고 생각한다.

이해를 기반으로 하고있다면 언젠가 사용할 날이 없더라도 크고 작은 그림을 그리는데 도움이 되는것 같았다.
