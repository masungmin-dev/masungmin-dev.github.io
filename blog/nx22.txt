21:T1f3f,<h1 id="nx-22-">Nx 22 훑어보기</h1><h2 id="">잡소리</h2><p>NX에 닷넷 섞어다 쓸 수 없을까 라는 생각이 문득 들었다.</p>
<p>이 이유에는 Nx가 21에서 @nx/gradle 을 발표한것도 있는데, Java 생태계를 끌고 올 수 있고 Javascript 프로젝트들과 함께 여러 언어에 대한 모노레포 환경을 구축할 수 있다고 생각하는 것 같다.(이 과정이 꽤 오래전부터 이어진 걸로 인지하고 있었는데, 지금 찾아보니 코로나 전 부터 생각이 있었나보다)</p>
<p>아무튼 그럼 닷넷도 가능하지 않을까 생각이 들었다.</p>
<p>내가 Nx를 접하고나서 좋아하게 된 이유 중 모노레포를 근사하게 만들어 낼 수 있다는 점도(이게 목표니까) 있지만 현업환경에서 필요했던 복잡한 자동화 과정을 excutor 로 풀어낼 수 있다는 점이었다.</p>
<p>사실 완벽한 조합은 안되더라도 쉘 실행 자체는 가능하니까 충분히 가능한 시나리오가 아닐까 라는 막연한 생각도 있었다.</p>
<blockquote>
<p>NX가 다른 모노레포와 달리 그 동안 Task 매니징에 대한 다양한 방법을 제공하기위해 끊임없는 관심을 갖고 있었고 관점을 바라보는 방식이 약간 달랐다고 느꼇는데, 그런 결과 이런 가능성과 욕심에 대해서도 진행해볼 수 있는게 (만들어 낼 수 있었던게) 아닐지... 그냥 개인적인 생각</p>
</blockquote>
<p>아무튼 찾다보니 이런 것도 있었다. 공식은 아닌듯하나 활발한 것 같아 보였음
<a href="https://github.com/nx-dotnet/nx-dotnet">https://github.com/nx-dotnet/nx-dotnet</a></p>
<p>그런데 무슨 우연인지 오늘 Nx22 릴리즈 공지에 닷넷이 있네..ㅎ (당황..)</p>

<p>내용을 읽다보니 아까 위에 있던 커뮤니티 플러그인 내용도 보인다.</p>
<blockquote>
<p>Our @nx/dotnet plugin started life as a community plugin named @nx-dotnet/core, and we recommend all users migrate to @nx/dotnet</p>
</blockquote>
<h3 id="pnpm-">pnpm 카탈로그 지원</h3><blockquote>
<p>PNPM 카탈로그란? : pnpm 카탈로그(Catalogs)는 pnpm v9.5.0부터 도입된 기능으로, 모노레포에서 여러 프로젝트 간에 의존성 버전을 중앙에서 관리할 수 있게 해주는 기능</p>
</blockquote>
<p>NX는 기존에도 개별 프로젝트에서 종속성을 관리하는 방법과 단일 버전을 루트에서 한번에 정의하던 방식을 모두 지원했다.</p>
<p>만약 개별 종속성 관리 방식이라면, pnpm 카탈로그 지원이 유용할 것 같다.</p>
<p>쉽게 생각하면 기존에 각각의 모노레포의 의존성을 pnpm 카탈로그로 단일 버전처럼 관리할 수 있는데, NX에서 이걸 지원한다고 보면 될 듯하다.</p>
<h3 id="nx-release-">NX release 개선</h3><p>nx에서 이제 백그라운드상에 ReleaseGraph를 구성한다고한다. 새로운 옵션(updateDependents:always)를 지원한다고 하는데, release 필터에 없는 종속중인 다른 프로젝트도 업데이트한다는 듯 하다.</p>
<p>아직 써본적 없지만 <a href="https://nx.dev/docs/guides/nx-release/update-dependents">이에 대한</a> 내용인듯.</p>
<p>그리고 스키마 개선, 사용자 정의 릴리즈 스크립트와 가이드에 대한 내용과 <strong>nx affected 계산 방식과 동일한 방식으로 release를 제어</strong>할 수 있다는 내용인것같다.</p>
<h3 id="ai-agent-configuration-">AI Agent configuration 개선</h3><p>claude code나 커서 등을 쉽게 구성할 수 있는 명령어를 추가한듯하다.</p>
<p>현업에서는 claude-code는 아직 보안상 이유로(컨펌안남..) 못 쓰고있는데 개인 플젝에는 써볼 수 있을 것 같다.</p>
<pre><code class="hljs language-bash">npx nx configure-ai-agents
</code></pre><h3 id="-">모듈 페더레이션</h3><p>nx 는 모듈페더레이션을 여지껏 잘 챙겨주고있는 듯 한데, 타입 안정성 개선, NX의 다양한 구성 방식에 대해 모든 호환 가능 등에 대한 내용인듯하다.</p>
<h3 id="--2">스토리북 지원 개선</h3><p>스토리북 플러그인에서 의존 관계를 자동으로 유추해서 스토리북이 다시 빌드되어야하는 상황을 더욱 잘 판단한다고 한다.</p>
<h3 id="-3">그외..</h3><p>몇가지 옵션들이 정리되거나 변경되었는데, 인상깊은건 이전에 예고한 createNodesV1(nx 플러그인 관련..)이 이제 날아간 듯하다. <del>이것 때문에 v2로 바꾸기위해.. 없어진 API 로 아이디어 내느라 고생했음..</del></p>
<h2 id="-4">마이그</h2><p>마이그레이션은 최근 nx 업데이트(몇버전이더라..)에서 vscode extension 을 활용한 UI 까지 생기면서 좀더 안정적으로 변했다고 판단하고 있었다. 그리고 한차례 오류가 많았던(커뮤니티에서도 불만의 목소리가 좀 보였던데..) 시기가 지나가서 nx21에서 올린다면 이번 변경사항들은 크게 마이그 과정에서 고통스럽진 않을 것 같다.</p>
<pre><code class="hljs language-bash">nx migrate --run-migrations
</code></pre><h2 id="--3">아무튼 멀티 언어 모노레포 환경.. 정말 기대해볼 수 있을까?</h2><p>자바나 닷넷 생태계에서 얼마나 관심을 가져줄 지 궁금하다. 혹은 그쪽 생태계에서는 현재 이런 거대해진 자바스크립트 생태계와 잘 통합할 수 있는 툴이 다른게 있을까? 닷넷을 현업으로 했던 시간이 이젠 조금 흐름만큼.. 혹시 있는지 궁금하기도 하다.</p>
<p>관심 가져주실지 모르겠으나 각각의 생태계에서 관심을 가져줫으면 좋겠다.</p>
<p>개인적으로 nx를 오래 써왓기 때문도 있지만 혹시 이런 업데이트 방향과 더불어 닷넷코어로 웹어셈블리를 만들고 JS 프로젝트에 손쉽게 통합할 수 있다거나 그런 사례를 만들 수 있다던지..</p>
<p>NextJS가 워낙 웹 풀스택으로 잘 나와 있지만 자바나 닷넷의 웹 프레임워크와 자바스크립트 프로젝트를 잘 연계할 수 있는...</p>
<p>아무튼 그런 태스크 매니징이나 의존성 그래프 관계를 맻을 수 있는 큰그림까지 가능한건지 궁금하고 기대된다.</p>
