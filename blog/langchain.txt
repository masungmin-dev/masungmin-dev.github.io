20:T2219,<h1>왜 갑자기 Langchain?</h1>
<p>최근 <a href="https://n8n.io/">n8n</a>으로 AI와 함께 워크플로우 자동화를 통해 이것저것 만들어보고 있습니다. 크론탭으로 배치를 돌려야하는 상황도 n8n으로 어느정도 커버가 가능한것같아서 개인적인 프로젝트를 진행할 때 데이터 적재를 꾀하기 아주 좋았습니다.</p>
<p>개인적으론 개인서버에 배치서버를 만들어서 배포하는게 번거롭기도하고 코드가 섞이는것도 싫어서... 또 구현하더라도 같은 API 서버로 두고싶지도 않았습니다.</p>
<p>그 와중에 n8n은 워크플로우를 프로젝트마다 생성해서 관리할 수 있으니 얼마나 편리한가? 싶었습니다.</p>
<p>그런데..</p>
<h2>n8n 쓰다보니 단점이 좀 보여요..</h2>
<p>n8n으로 &quot;AI와 함께하는 자동화된 워크플로우&quot;를 꾀하거나 혹은 &quot;AI를 활용한 어플리케이션처럼 보이는 무언가&quot;를 만들 수 있어보입니다.</p>
<p>하지만 어플리케이션 레이어에서 프로덕션 레벨로 쓸만한 역할로 만들어내기엔 목적과 성향이 좀 다르다 보입니다. 또 그런 레이어를 위한 용도는 아닌것 같았고 결국 LangChain 같은 프레임워크로 AI Agent 를 구축하는것도 공부하는게 좋아보였습니다.</p>
<p>그리고..</p>
<ul>
<li>디버깅이 GUI 로 하려다보니 영 불편하다(핀이 있긴한데.. 그래도 좀..)</li>
<li>아무래도 자동화 워크플로우용이니까...이거로 할 수 있는게 제한적인게 보이기 시작한다.</li>
<li>프로토타이핑 하기엔 좋은데.. 노드 중간에 데이터 가공과정을 노드에서 하려니 좀..손이 좀 많이 가네..?</li>
<li>개발자라 그런가.. 이거 그냥 GUI보다 코드로 보는게 더 가독성이 좋겠는데 ㅠ</li>
</ul>
<blockquote>
<p><em>그리고 개발자라면 특히나 뭔가 내 어플리케이션이라는 느낌이 덜 듭니다. github actions나 jenkins를 다루는 느낌이 강하게 느껴집니다.. 나는 내새꾸가 만들고싶다. 자꾸 개발자스런 욕망이 듭니다</em></p>
</blockquote>
<p>또 n8n self hosted 버전에서는 제한적인 요소들이 조금씩 보이기 시작합니다. 이해는 하지만 확실히 자유도는 떨어집니다.</p>
<p>그래서 n8n말고 결국 langchain 눈에 들어오기 시작했습니다.</p>
<h1>그래서 LangChain 은 뭔데?</h1>
<p>일단 n8n이나 그동안 AI에 대해 공부하면서 얻은 사전 지식이 있어서 처음 langchain 에 대해 알게 되었을 때보단 의욕도 생기고 이해가 쉬워졌습니다.</p>
<p>기본적으로 langchain하면 대표적인 것들이</p>
<ol>
<li>프롬프트의 관리(템플릿화하고 재사용 가능한 설계)</li>
<li>체인(순차적으로 연결, 요약-&gt;번역-&gt;저장 따위의 가공행위나 프로세스를 연결)</li>
<li>대화 히스토리 따위를 저장, 맥락(context)유지 가능</li>
<li>Agent화, 도구를 선택해서 사용하도록 지시 가능</li>
<li>문서 가져오기, DB든 파일이든 등등</li>
<li>벡터 스토어와 RAG</li>
</ol>
<p>이런 요소들인데, 이를 활용하면 어쨋건 AI 로 할수있는 단순한 수준의 무언가는 만들 수 있겠다 라는 생각이 듭니다. (설명보단 공부 후 느낀점 일기 위주다보니 많이 생략됨..)</p>
<p>그리고 Python과 Typescript(javascript)를 지원하고 있는데, TS 로 나온 예시보단 파이썬 예시가 더 많은편 같습니다. 그래도 일단 node 개발자로서 TS를 고집해서 공부해보고 있습니다 (ㅋㅋ)</p>
<p>아무튼 레이어로 표현하면 대강 이런 그림인것 같습니다.</p>

└─────────────────────────────────────────────┘
</code></pre><p>어플리케이션은 (높은확률로) API가 될것이고, API 에서 클라 레이어에 어울릴 통신 방식으로 제공해서 잘 꾸며서 넣으면... 그림이 나올것같습니다. 실제 실무레벨에선 어떨진 모르겠습니다. 아무튼 아직까지 느낀바는 이렇습니다. 그렇게 활용하지 않을지?</p>
<p>그리고 LLM에서 RAG까지 가져가야 할테니.. 임베딩모델이라던가 등등이 또 필요할거고 결국 서비스에 어울리는 필요한 데이터의 적재와 가공 과정의 레이어는 또 별도로 필요할 것 같습니다. 요청이 올떄마다 할 순 없으니까.</p>
<p><em>아아.. 얼마나 재미있는 서비스들이 많이 나올 수 있을지 인프라와 돈, 시간만 있다면 이것저것 만들고싶습니다. 막상 또 만들기 시작하면 고뇌로 힘들겟지만..(취미로 개발하는 로또 당첨된 개발자가 하고싶다)</em></p>
<h3>RAG</h3>
<p>사실 AI 관련 서비스를 만들어보니까 RAG가 제일 궁금(필요)했는데 나중에 따로 좀 글로 남겨야겠습니다. 관심있게 보다보니 RAG도 다양한 개선 방법과 접근방법이 있어서 몇몇 찾아보고 공부해보았는데 또 뭐가 많을 것 같습니다. 나중에 복습하기위해 링크도 좀 모아두고 접근법에 대해 적어두고 싶어졌습니다. 어쨋건 AI 서비스 고도화를 위한 키 중 하나이고 어떻게 설계하느냐에 따라서 비용과 퀄리티가 굉장히 다른것같습니다.</p>
<h3>LangGraph와 그외 등등등..</h3>
<p>처음 이야기했던 n8n 처럼 활용하려면 LangGraph도 잘 활용할 수 있어야할 것 같습니다.</p>
<p>LangGraph가 있으면 n8n처럼 여러형태의 플로우를 만들 수 있고, 비로소 n8n처럼 활용할 수 있는 것 같습니다. 그렇게 된다면 더 자유로운 AI워크플로우를 만들 수 있을 것이며, 어플리케이션 레벨의 무언가를 만드는데 더 공격적인 무기로 활용될 것 같습니다.</p>
<h2>아무튼..</h2>
<p>랭체인으로도 충분히 많은걸 할 수 있고 알겠는데 조금 더 만들고자하는 AI 서비스를 (컴포넌트화 하고 체계적으로 관리하고 플로우를 잘 만들거나 Mutli Agent를 구현한다던지)고도화하려면 LangGraph도 필수인것같습니다. 공부하다보니(+코드로 구현해보다보니) 확실히 LangGraph가 있으니 좀더 추상적이고 관리가능한 노드 단위로 LangChain으로 만든것들을 분류하고 구성하는 그림이 보여지고 그려지기 시작했습니다.</p>
<p>그외 아직 궁금한 키워드들이 많습니다. 이거로 토이프로젝트나 업무에서 고도화 할 수 있는 요소들이 많아질 것 같아 기대되네요. 아무튼 AI 덕분에 서비스를 만드는데 새로운 풍부한 무언가가 생겨서 좋은것같습니다.</p>
