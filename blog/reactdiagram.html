React의 API들을 좀더 쉽게 연관짓기

2025년 10월 23일

왜 이런 이야기를 하는건가?

오늘 집에가다 든 생각이다.

프론트엔드 개발자가 하루종일 React만 쓰는 것은 아니고, 물론 계속해서 따라가면 좋겠지만 모든 API 들을 적절한 상황 속에서 계속 실무에 팔로업하기는 어려울 수 있다고 생각한다. 이걸 어떻게 쉽게 설명할 수 있고 연관관계를 상상할 수 있게 만들 수 있을까?

그래도 시간 투자하면 따라갈 순 있지만 근본적으로 점점 기억해야할게 많아지고 API간의 관계성이 많아지고 있다.

어렵게 느껴지는 이유

그 이유는 개인적으로 너무 '매지컬'하게 느끼기 떄문이라고 생각한다.

우리는 항상 최신 패키지를 사용하는 것 또한 아니고, 관심있게 모두 팔로업하면 좋겠지만 현업과 일상 생활에서 중요도에 따라 밀리기도 한다.

예전에는 단순히 라이프사이클, 렌더링을 어떻게 최적화 했는지 정도였지만 이제는 다양한 피쳐들의 (매지컬한)상호관계 떄문에 그렇게 느끼는 것 같다. 그리고 후술하겠지만 어떤 이유로 API 도 점점 많아진다.

매지컬하면 쉬운데 어려워지는 이유

이렇게 작성하면 저렇게 동작한다 라는 것 자체를 쫓아가는 것도 힘들어질 수 있다.

혹은 잘 모르지만 그렇게 동작하더라 식 개발도 가능해진다. 마치 내가 개발을 처음 접했을 때 .NET Framework MVC를 마주했을 때 그런 느낌이었다.

.NET Framework의 .NET MVC는 Next.js처럼 파일 시스템 구조를 기반으로 라우팅을 하는 등, 많은 부분을 위임해서 직접 해주고 있었다.(커스텀 등등 가능했지만) 매지컬하게 느낀 이런 부분들이 개발할 땐 정말 편해도 잘 몰라도 잘 되니까 더더욱 게을러지고 그냥 넘기게 되었던 부분이 오히려 나의 찾아보려는, 이해하려는 욕구를 막고 안좋은 버릇이 들었을 때도 있었던 것 같다.

React도 마찬가지인 것 같다. 엣지 포인트의 사이사이에 API를 추가해주는건 좋지만 혹시 러닝커브가 너무 올라가는게 아닐까? 여러 API들과 React.Suspense의 상호 관계도 그렇게 보이진 않을까? 라는 생각이 들기 시작했다.(아니면 다행) React Compiler의 메모라이징 자동화도 앞으로 누군가에겐 매지컬하게 보일 수도 있을 것 같다.

매지컬한게 나쁜건 아니라고 생각하는데(오히려 좋은 점이 많지만) 어쨋든 상호관계를 잘 기억하고 머릿속에 정리하고있으며 이해를 하고 있길 원하는게 나도, 나를 평가하는 사람도, 내 동료들도 원하는바가 아닐까 싶다.

아무튼 이제는 동작 방식에 대한 이해를 하고있느냐도 좋지만 상호 관계를 잘 기억해 두어야 쓸 수 있는 요소들이 있다.

아무튼 그래서 집에가다 든 생각이..

잡소리가 너무 긴거같은데...아무튼.. 이런 생각을 하고서 해보고싶었던게..

이걸 브레인 스토밍 하는 것처럼 서로 연결짓는 방식으로 해두면 좀더 이해하기(혹은 훑기) 쉽지 않을까 싶었다.

근데 사실 React 현업자라면 대부분 어느정도 연관짓기 쉽다. 그냥 해보고싶었음

그래도 다시 한번 내가 아는 것을 의심하고자.. 한번 AI 딥리서치로 만들어보면 어떨까 싶어서 오자마자 지금 그려보았는데..

아니 다 해달라고 하니까 너무 길다 ㅋㅋ 그럼 좀 잘라서 문서화 해달라고 요청해보자 (오푸스야 힘내렴)

그래서 정리된 버전

각각 좀 쪼개서 달라고 하니까 주절주절 수다를 적다보니 딥리서치에서 나온걸 복붙해본다.

정말 내용이 맞는지 스스로 확인해보면서 공부하면 좋을 것 같다.

혹은 더 나아가서 스스로 한번 그려보는 것도, 도움이 될 것 같다. 스터디하면 가끔 브레인스토밍으로 머리 속에 부족한 그레이 영역을 찾아낼 수 있어서 항상 효과가 좋았던 것 같다.

이참에 나도 다시한번 들여다보길, vscode로는 마크다운 프리뷰로 봐도 약간 보기 힘들어서 올려놓고 봐야지 싶다.

전체 API 구조도(간소화)

1. 상태관리

  • useState → useReducer: 상태 로직이 복잡해지면 useReducer로 전환
  • useState → useOptimistic: 서버 응답 전 UI를 미리 업데이트
  • Server Action → useActionState: 서버 액션의 pending, error, data 상태 관리
  • useSyncExternalStore: Redux 등 외부 저장소와 React 동기화

2. Context

값 변경 시 모든 구독자가 리렌더링됨

  • createContext → Context: Context 객체 생성
  • Provider → useContext: Hook을 통한 값 소비 (권장)
  • Provider → Consumer: Render prop 패턴 (레거시)
  • use Hook: React 19의 새로운 Context 읽기 방식

3. Suspense & Async (비동기 처리)

  • lazy → Suspense: 동적 임포트된 컴포넌트 로딩
  • use → Suspense: Promise를 throw하여 Suspense 트리거
  • Suspense → ErrorBoundary: 에러 발생 시 가장 가까운 ErrorBoundary로 전파
  • useTransition: Suspense와 함께 사용하여 로딩 상태 제어

4. Ref System (Ref 관리)

  • useRef: DOM 접근 + 렌더링과 무관한 값 저장
  • forwardRef: 컴포넌트가 ref를 받을 수 있도록 함
  • useImperativeHandle: ref를 통해 노출할 인스턴스 값 커스터마이징
  • 부모가 자식의 메서드를 호출할 수 있게 함

5. Performance Optimization (성능 최적화)

  • React.memo: props가 변경되지 않으면 리렌더링 스킵
  • useMemo: 비싼 계산의 결과를 메모이제이션
  • useCallback: 함수 참조를 유지하여 불필요한 리렌더링 방지
  • useCallback + memo 조합: 자식 컴포넌트 리렌더링 최적화

6. Effects System (부수 효과)

  • useEffect: Paint 이후 비동기로 실행 (대부분의 경우 사용)
  • useLayoutEffect: Paint 이전 동기로 실행 (DOM 측정 필요시)
  • useInsertionEffect: 모든 DOM 변경 이전 (CSS-in-JS 라이브러리용)
  • 모든 Effect는 클린업 함수 반환 가능

7. Concurrent Features (동시성 기능)

  • useTransition: 상태 업데이트를 논블로킹으로 처리
  • startTransition: Hook 외부에서 사용하는 명령형 버전
  • useDeferredValue: 값의 업데이트를 지연시켜 UI 반응성 유지
  • isPending: 트랜지션 진행 중인지 확인

8. Server Components & Actions (React 19)

  • Server Component: 서버에서 실행, 직접 데이터베이스 접근 가능
  • Server Action: 'use server' 지시어로 서버에서 실행되는 함수
  • useActionState: Server Action의 상태(pending, error, data) 관리
  • useOptimistic: 서버 응답 전 UI를 낙관적으로 업데이트
